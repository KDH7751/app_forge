# AI Workflow

## 목적

이 문서는 AI와 협업할 때의 작업 순서와 출력 기준을 고정한다.

## 작업 전 확인

1. 현재 코드와 문서를 먼저 확인한다.
2. 현재 구조와 충돌하는 지점이 있으면 먼저 짚는다.
3. public API 또는 public entry를 먼저 정의하거나 확인한다.
4. format, analyze, test로 검증한다.
5. 문서와 코드 기준이 다르면 함께 동기화한다.

## 구현 전 우선 응답

구조가 바뀌는 작업은 바로 코드부터 쓰지 않는다.
먼저 아래를 보여줘야 한다.

- 파일 트리
- public entry 또는 주요 표면 변경점
- 충돌 지점 또는 애매한 점
- 구현 범위

## 구현 원칙

- 한 번에 큰 범위를 모두 구현하지 않는다.
- 기존 코드와 문서를 무시하고 새로 덮어쓰지 않는다.
- 구조와 계약을 확인한 뒤 구현에 들어간다.
- 현재 Phase 범위를 넘는 구현은 하지 않는다.
- 하나의 변경 의도와 책임으로 자연스럽게 묶이는 범위로 수정한다.

## 필수 출력

AI는 매 응답에서 최소한 다음을 포함해야 한다.

- 변경 의도
- 생성/수정 파일 목록
- 파일별 한 줄 책임
- 내가 확인할 체크포인트

## 금지 패턴

- UI에서 Firebase를 직접 호출하는 것
- Engine이 app이나 Feature를 import하는 것
- raw throw 기반 async 흐름을 그대로 유지하는 것
- 현재 Phase 범위를 넘는 기능을 몰래 추가하는 것
- 하나의 변경 의도와 무관한 파일을 불필요하게 넓게 수정하는 것

## 완료 기준

작업은 코드 작성으로 끝나지 않는다.
최소한 아래가 확인되어야 한다.

- 구조 규칙을 지켰는지
- analyze가 통과하는지
- test가 통과하는지
- 이번 작업 목표를 충족했는지

## Commit Message

커밋 메시지는 일관된 형식을 유지한다.

기본 형식:

```bash
git commit -m "<type>: <요약>" -m "<상세 설명>"
```

### type

다음 중 하나를 사용한다.

- `feat`: 기능 추가 또는 Phase 진행
- `refactor`: 구조 변경 또는 코드 재정리
- `docs`: 문서 또는 주석 변경
- `fix`: 버그 수정
- `chore`: 설정, 빌드, 환경 정리

### 작성 기준

- 요약은 한 줄로 작성하며, 커밋 메시지만 보고도 변경 의도를 이해할 수 있어야 한다.
- 최근 커밋 제목은 `type:` 뒤에 한글 설명을 두고, `phase`, 기술명, 모듈명만 영어로 섞어 쓰는 형식을 기본으로 한다.
- 변경 범위가 중요하면 `phase`, `app`, `engine`, `feature`, `docs` 같은 맥락을 드러내고, 제목과 본문을 영어만 사용해 일괄 작성하지 않는다.
- `정리`, `반영`, `갱신` 같은 추상 표현만으로 제목을 닫지 않고, 무엇을 바꿨는지 드러나는 구체 명사를 포함한다.

### 상세 설명

- 상세 설명은 한글 bullet list를 기본으로 하고, 타입명이나 모듈명 같은 식별자만 영어로 유지하며, 명사형으로 끝낸다.
- 파일 이름 나열보다 변경 의도 단위로 묶어 설명하고, 구조 변경, 계약 변경, 문서 보완처럼 성격이 다른 변경은 항목을 나눠 적는다.
- 사소한 변경이 아니면 상세 설명을 생략하지 않는다.
