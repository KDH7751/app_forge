# AI Workflow

## 목적

이 문서는 AI와 협업할 때의 작업 순서와 출력 기준을 고정한다.

## 작업 전 확인

1. 현재 코드와 문서를 먼저 확인한다.
2. 현재 구조와 충돌하는 지점이 있으면 먼저 짚는다.
3. public API를 먼저 정의하거나 확인한다.
4. format, analyze, test로 검증한다.

## 구현 전 우선 응답

구조가 바뀌는 작업은 바로 코드부터 쓰지 않는다.

먼저 아래를 보여줘야 한다.

- 파일 트리
- 주요 타입 / public API
- 충돌 지점 또는 애매한 점
- 구현 순서

## 구현 원칙

- 한 번에 큰 범위를 모두 구현하지 않는다.
- 기존 코드와 문서를 무시하고 새로 덮어쓰지 않는다.
- 구조와 계약을 확인한 뒤 구현에 들어간다.
- 현재 Phase 범위를 넘는 구현은 하지 않는다.
- 하나의 구조 변경이나 기능 흐름을 자연스럽게 닫기 위해 필요하다면 여러 파일을 한 번에 수정할 수 있다.
- 중요한 것은 파일 수가 아니라, 한 번의 변경이 하나의 목적과 책임으로 묶여 있는가이다.

## 필수 출력

AI는 매 응답에서 최소한 다음을 포함해야 한다.

- 변경 의도
- 생성/수정 파일 목록
- 파일별 한 줄 책임
- 내가 확인할 체크포인트

## 금지 패턴

- UI에서 Firebase를 직접 호출하는 것
- Engine이 app이나 Feature를 import하는 것
- raw throw 기반 async 흐름을 그대로 유지하는 것
- 현재 Phase 범위를 넘는 기능을 몰래 추가하는 것
- 하나의 변경 의도와 무관한 파일을 불필요하게 넓게 수정하는 것

## 완료 기준

작업은 코드 작성으로 끝나지 않는다.

최소한 아래가 확인되어야 한다.

- 구조 규칙을 지켰는지
- analyze가 통과하는지
- test가 통과하는지
- 이번 Phase 목표를 충족했는지

## Commit Message 규칙

커밋 메시지는 일관된 형식을 유지하며,
사람과 AI 모두 동일한 기준으로 작성한다.

기본 형식:

```bash
git commit -m "<type>: <요약>" -m "<상세 설명>"
```

### 1. type

다음 중 하나를 사용한다.

- `feat`: 기능 추가 또는 Phase 진행
- `refactor`: 구조 변경 또는 코드 재정리 (동작 변화 없음)
- `docs`: 문서 / 주석 변경
- `fix`: 버그 수정
- `chore`: 설정 / 빌드 / 환경 정리

### 2. 작성 절차

커밋 메시지는 다음 순서로 작성한다.
AI도 동일한 절차를 따른다.

#### 1) 변경 범위 파악

- `git add` 된 파일을 기준으로 변경 범위를 판단한다.
- 파일 경로를 기반으로 engine / app / feature / docs 영역을 구분한다.

#### 2) 변경 의도 결정

- 기능 추가 → `feat`
- 구조 변경 → `refactor`
- 문서/주석 → `docs`
- 버그 수정 → `fix`
- 설정/환경 → `chore`

여러 성격이 섞이면 가장 핵심 의도를 기준으로 선택한다.

#### 3) 요약 작성

- 한 줄로 작성한다.
- 무엇을 했는지 중심으로 표현한다.
- 변경 범위(engine/app/feature 등)가 드러나면 좋다.
- 필요하면 Phase를 포함한다.

#### 4) 상세 설명 작성

- bullet list로 작성한다.
- 변경 내용을 충분히 설명한다.
- 파일 단위가 아니라 변경 의도 단위로 묶는다.
- 단순 파일 나열은 하지 않는다.

### 3. 스타일 규칙

- 기존 커밋의 설명적인 스타일을 유지한다.
- 문장을 축약하지 않는다.
- 커밋만 보고 변경 의도를 이해할 수 있어야 한다.
