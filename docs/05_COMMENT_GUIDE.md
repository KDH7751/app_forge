# Comment Guide

## 목적

주석은 코드를 번역하지 않는다.
주석은 코드만으로 드러나지 않는 구조, 경계, 제약, 의도를 설명한다.

## 기본 원칙

- 무엇을 하는지보다 왜 이렇게 설계했는지를 설명한다.
- 코드로 충분히 드러나는 내용은 주석으로 반복하지 않는다.
- 오래 유지될 정보만 남긴다.

## 언제 주석을 쓰는가

다음 경우에만 주석을 쓴다.

- 구조적 역할이 중요한 경우
- 의존성 경계가 중요한 경우
- 의도적으로 제한한 정책이 있는 경우
- 외부 계약이나 실패 조건이 중요한 경우
- 코드만 보면 오해하기 쉬운 경우

## 파일 상단 주석 규칙

모든 파일에 같은 길이의 주석을 강제하지 않는다.

### 강한 헤더가 필요한 파일

다음처럼 구조적으로 중요한 파일에는 강한 헤더를 둔다.

- Engine contract / core file
- Router / Shell 핵심 파일
- composition root
- repository / datasource 핵심 파일
- provider / controller 핵심 파일

예:

```dart
/// ===================================================================
/// RouterEngine
///
/// 역할:
/// - RouteDef tree를 GoRouter로 변환한다.
///
/// 경계:
/// - auth policy를 직접 알지 않는다.
/// - navigation sync만 담당한다.
/// ===================================================================
```

### 간략 주석이면 충분한 파일

다음처럼 의미가 비교적 단순한 파일은 짧은 주석으로 충분하다.

- page, widget, dto, mapper, 단순 state holder

예:

```dart
/// posts 상세 화면을 렌더링한다.
class PostDetailPage extends ConsumerWidget { ... }
```

또는 구조 설명이 필요 없다면 파일 상단 주석을 생략할 수 있다.

## Public API 주석 규칙

모든 public API에 장문 주석을 강제하지 않는다. 다음 경우에만 주석을 단다.

- 외부 계약이 중요한 경우
- 실패/제약 조건이 중요한 경우
- side effect가 있는 경우
- 이름만으로 의도가 충분히 드러나지 않는 경우

짧은 예:

```dart
/// 현재 location에서 RouteDef를 찾는다.
RouteMatch? matchRoute(...)
```

조금 더 강한 예:

```dart
/// login을 수행한다.
///
/// 실패는 외부로 throw하지 않고 AppError로 반환해야 한다.
Future<Result<User>> login(...)
```

## 금지 주석

- 코드를 그대로 읽어주는 주석
- 자명한 동작 설명
- TODO / FIXME 남발
- 왜가 없는 주석
- 코드와 어긋난 오래된 주석

잘못된 예:

```dart
// 버튼을 누르면 이동한다.
onPressed: () { ... }
```

잘된 예:

```dart
/// 상세 화면은 shell 안에 두되,
/// bottomNav는 route metadata로 숨긴다.
```

## 주석 언어 규칙

- 코드 식별자 관련 용어는 영어로 유지한다.
- 설명 문장은 한글로 작성한다.
- 구조와 아키텍처 키워드는 영어로 유지한다.

## 한 줄 원칙

주석은 필요한 곳에만 짧고 강하게 쓴다.
