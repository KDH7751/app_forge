# Comment Guide

## 목적

주석은 코드를 번역하지 않는다.
주석은 코드만으로 드러나지 않는 구조, 경계, 제약, 의도를 설명한다.

## 기본 원칙

- 무엇을 하는지보다 왜 이렇게 설계했는지를 설명한다.
- 코드로 충분히 드러나는 내용은 주석으로 반복하지 않는다.
- 오래 유지될 정보만 남긴다.

## 파일 상단 주석 규칙

모든 파일에 같은 길이의 주석을 강제하지 않는다.

### 강한 헤더가 필요한 파일

다음처럼 구조적으로 중요한 파일에는 강한 헤더를 둔다.

- Engine contract / core file
- Router / Shell 핵심 파일
- composition root
- repository / datasource 핵심 파일
- provider / controller 핵심 파일

강한 헤더는 `역할:` / `경계:` 라벨형을 기본으로 사용한다.

예:

```dart
/// ===================================================================
/// RouterEngine
///
/// 역할:
/// - RouteDef tree를 GoRouter로 변환한다.
///
/// 경계:
/// - auth policy를 직접 알지 않는다.
/// - navigation sync만 담당한다.
/// ===================================================================
```

### 간략 주석이면 충분한 파일

다음처럼 의미가 비교적 단순한 파일은 짧은 주석으로 충분하다.

- page, widget, dto, mapper, 단순 state holder

예:

```dart
/// posts 상세 화면.
class PostDetailPage extends ConsumerWidget { ... }
```

또는 구조 설명이 필요 없다면 파일 상단 주석을 생략할 수 있다.

## 클래스 / 함수 주석 규칙

- 클래스와 함수에는 짧게라도 한 줄 주석을 둔다.
- 클래스는 역할이나 경계를 짧게 설명한다.
- 함수는 무엇을 위한 함수인지, 필요한 경우 제약이나 의도를 짧게 설명한다.
- 장문 계약 주석은 꼭 필요한 경우에만 쓴다.

예:

```dart
/// 현재 location에서 RouteDef를 찾는다.
RouteMatch? matchRoute(...)

/// login 수행.
///
/// 실패는 외부로 throw하지 않고 AppError로 반환해야 한다.
Future<Result<User>> login(...)
```

## 변수 / 필드 주석 규칙

- 변수와 필드는 기본적으로 주석을 생략할 수 있다.
- 다만 역할이 비슷한 값이 함께 존재해 구분이 필요하면 메모형 주석을 짧게 둔다.

예: 

```dart
/// Router tree 구성용 top-level route 목록.
final appRouteTrees = ...

/// matcher / currentRoute 판별용 flat route 목록.
final appRoutes = ...
```

## 짧은 주석 톤 규칙
- 짧은 설명 주석은 메모형 또는 짧은 설명형을 선호한다.
- 불필요하게 ~이다., ~한다. 형태로 늘이지 않는다.
- 파일 헤더와 짧은 설명 주석의 톤은 같을 필요가 없다.

## 금지 주석

- 코드를 그대로 읽어주는 주석
- 자명한 동작 설명
- TODO / FIXME 남발
- 왜가 없는 주석
- 코드와 어긋난 오래된 주석

## 주석 언어 규칙

- 코드 식별자 관련 용어는 영어로 유지한다.
- 설명 문장은 한글로 작성한다.
- 구조와 아키텍처 키워드는 영어로 유지한다.

## 한 줄 원칙

클래스와 함수는 짧게라도 설명하고, 그 외는 필요한 곳에만 짧고 강하게 쓴다.
