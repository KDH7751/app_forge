# Comment Guide

## 목적

주석은 코드를 장식하는 문장이 아니라, 코드 독자가 선언의 실제 역할을 바로 이해하게 만드는 설명이다.

이 가이드는 주석을 단순히 줄이기 위한 규칙이 아니다.
상세하더라도 읽는 즉시 도움이 되는 주석을 남기기 위한 기준을 정한다.

주석은 최소한 다음 역할을 해야 한다.

- 이 선언이 무엇을 하는지 바로 알 수 있어야 한다.
- 이 선언이 어디에 쓰이는지 알 수 있어야 한다.
- 수정 시 어떤 영향이 생기는지 알 수 있어야 한다.
- docs 없이 코드만 봐도 최소한의 맥락을 이해할 수 있어야 한다.

## 기본 원칙

- 코드 이름을 번역하지 말고, 실제 쓰임새와 연결 지점을 설명한다.
- docs의 구조 설명을 복사하지 말고, 이 선언을 읽는 사람에게 필요한 사용 설명을 적는다.
- 상세하게 써도 되지만, 읽고 나서 바로 행동할 수 있는 정보여야 한다.
- 오래 유지될 정보만 남기되, 수정 영향이 큰 선언은 그 영향 범위를 분명히 적는다.

## 주석 밀도 레벨

모든 코드에 동일한 수준의 상세 주석을 요구하지 않는다.

코드의 역할과 위치에 따라 주석 밀도를 다르게 적용한다.

### Level 1 — 강한 설명 필요

다음처럼 구조적으로 중요하거나 수정 영향이 큰 파일에는 상세한 주석을 둔다.

- composition root (app_config.dart, app_features.dart, app_plugins.dart)
- engine core (ErrorHub, RouterEngine 등)
- policy / decision / contract
- redirect / routing 핵심 로직
- plugin wiring
- repository / datasource 핵심 파일

Engine 내부에서는 다음 조건을 만족하면 Level 1로 본다.

- app 전체 동작에 직접적인 영향을 주는 흐름을 제어하는 경우
- app에서 override하거나 주입되는 contract인 경우
- 여러 feature에서 공통으로 사용되는 core abstraction인 경우

예:

- ErrorHub
- ErrorPolicy / DefaultErrorPolicy
- RouterEngine
- Shell / routing core

composition root(app)에서는 Level 1이라도 작성 방식이 다르다.

app 레이어는 flow나 policy를 직접 구현하는 곳이 아니라,
Engine과 Feature를 조립하고 app 입력을 정의하는 위치다.

따라서 app 주석은 다음 기준을 따른다.

- 내부 동작 설명보다 이 값이 app 전체에 어떤 영향을 주는지 설명한다
- Engine 구조나 내부 흐름을 다시 설명하지 않는다
- docs에 있는 구조 설명을 반복하지 않는다
- 길게 쓰기보다 영향 중심으로 2~4줄 안에서 정리한다

예:

- 이 설정을 바꾸면 app 전체 에러 처리 방식이 달라진다
- 이 feature 목록에 따라 app 전체 라우팅 구조가 달라진다
- 이 plugin 조합에 따라 외부 연동 방식이 달라진다

bootstrap 레이어는 runtime host로서 engine과 app을 연결하는 역할을 가진다.

따라서 bootstrap 주석은 다음 기준을 따른다.

- 내부 구현보다 "어떤 흐름을 언제 연결하는지"를 설명한다
- 실행 순서와 초기화 시점이 드러나야 한다
- 이 연결을 바꾸면 앱 전역 실행 흐름에 어떤 영향이 있는지 설명한다
- engine 내부 구조나 app 설정 내용을 다시 설명하지 않는다

이 경우에는 선언 단위 주석도 더 상세하게 작성할 수 있다.
파일 헤더 작성 방식은 아래 "파일 상단 주석 규칙"을 따른다.

---

### Level 2 — 기본 설명 (기본값)

다음처럼 일반적인 로직 파일은 간결한 설명을 기본으로 한다.

- controller
- provider
- repository
- mapper
- helper
- 일반 state 흐름 코드

이 경우:

- 1~3줄 정도의 설명을 기본으로 한다
- "무엇을 하는지"와 "어디서 쓰이는지" 정도만 포함한다
- 수정 영향이 특별히 크지 않다면 상세한 영향 설명은 생략할 수 있다

---

### Level 3 — 최소 설명

다음처럼 역할이 비교적 단순하고 이름만으로도 이해 가능한 코드는 최소 설명만 둔다.

- UI widget
- DTO
- 단순 state holder
- 단순 model / enum

이 경우:

- 이름만으로 충분하면 주석을 생략할 수 있다
- 필요할 때만 한 줄 설명을 추가한다

## docs와 Code Comment의 역할 분리

- docs는 구조, 철학, layer 규칙, 설계 결정, 큰 흐름을 설명한다.
- code comment는 파일/선언을 바로 읽는 사람을 위한 사용 설명이어야 한다.
- 같은 내용을 docs와 code comment에 모두 적을 수는 있다.
- 다만 허용되는 경우는, 그 설명이 해당 선언을 이해하는 데 직접 필요할 때뿐이다.

예를 들어 docs에는 "app이 redirect policy를 소유한다"를 적을 수 있다.
하지만 code comment에는 "이 함수가 인증 상태와 현재 location을 기준으로 redirect 대상을 결정한다"처럼
해당 선언을 바로 읽는 사람에게 필요한 설명으로 다시 써야 한다.

## 문맥 기반 주석 판단

같은 형태의 주석이라도, 코드의 위치와 역할에 따라
좋은 주석이 될 수도 있고 부족한 주석이 될 수도 있다.

예:

```dart
/// app 활성 Feature 등록 목록.
final appFeatures = <EngineFeature>[...];
```

이 문장은 일반적인 데이터 변수나 단순한 목록에는 충분히 유효한 주석이다.

하지만 composition root에서는 부족한 주석이다.
이 위치에서는 다음 정보가 추가로 드러나야 한다.

- 이 값이 어디에서 소비되는가
- 이 값이 앱 전체 동작에 어떤 영향을 주는가

반대로 단순한 데이터나 지역 상태에서는
간단한 역할 설명만으로 충분하다.

즉, 주석의 적절성은 문장 자체가 아니라
"이 코드가 어떤 역할을 하는 위치인가"를 기준으로 판단한다.

## 파일 상단 주석 규칙

모든 파일에 같은 길이의 주석을 강제하지 않는다.

### 강한 헤더가 필요한 파일

다음처럼 Level 1에 해당하는 구조적 중심 파일에는 강한 헤더를 둔다.

- Engine contract / core file
- Router / Shell 핵심 파일
- composition root
- repository / datasource 핵심 파일
- app 전역 흐름을 조정하는 핵심 provider / controller

강한 헤더는 파일을 열자마자 다음이 보이도록 쓴다.

- 이 파일이 담당하는 핵심 기능
- 이 파일에서 결정되는 것
- 이 파일을 수정할 때 조심할 점

라벨형을 쓴다면 `역할:` / `결정:` / `주의:` 또는 이에 준하는 구성을 기본으로 사용한다.

예:

```dart
/// ===================================================================
/// RouterEngine
///
/// 역할:
/// - RouteDef tree를 GoRouter로 변환한다.
///
/// 결정:
/// - route tree가 실제 router 설정으로 바뀌는 방식이 여기서 정해진다.
///
/// 주의:
/// - auth policy를 직접 알지 않는다.
/// - navigation sync 범위를 넘는 책임을 넣지 않는다.
/// ===================================================================
```

### 간략 주석이면 충분한 파일

다음처럼 의미가 비교적 단순한 파일은 짧은 주석으로 충분하다.

- page, widget, dto, mapper, 단순 state holder

예:

```dart
/// 특정 post의 상세 내용을 보여주는 화면.
class PostDetailPage extends ConsumerWidget { ... }
```

또는 구조 설명이 필요 없다면 파일 상단 주석을 생략할 수 있다.

## 클래스 / 함수 주석 규칙

- 클래스와 함수는 이름만으로 역할이 충분하지 않으면 주석을 둔다.
- 구조적으로 중요하거나 수정 영향이 큰 경우에는 짧게라도 설명한다.
- 기준은 "짧게"가 아니라 "읽자마자 역할이 이해되는가"다.
- 클래스는 어떤 책임을 가지는지, 어디에서 소비되는지, 수정 영향이 크면 무엇이 함께 바뀌는지 적는다.
- 함수는 무엇을 입력으로 받아 어떤 결정을 내리는지, 어디에 쓰이는지, 호출 시 기대되는 효과를 적는다.
- 이름만으로 충분히 안 보이면 여러 줄 주석을 써도 된다.

특히 다음 선언은 이름만으로 직관적이지 않으면 주석을 보강한다.

- mapper
- policy
- provider
- helper
- aggregation 함수
- route / redirect 관련 함수
- plugin 초기화 함수

예:

```dart
/// 현재 location에 대응하는 RouteDef를 찾는다.
///
/// router sync와 shell route 판별에서 공통으로 사용된다.
RouteMatch? matchRoute(...)

/// login 요청을 수행한다.
///
/// UI에서는 이 결과만 소비하므로,
/// 실패는 외부로 throw하지 않고 AppError로 반환해야 한다.
Future<Result<User>> login(...)
```

## 변수 / 필드 주석 규칙

- 변수와 필드도 역할이 즉시 보이지 않으면 주석을 둔다.
- 특히 composition root 값, policy 조합, route 집계 결과, mapper 목록처럼
  "앱에서 실제로 어떤 입력을 쓰는가"를 나타내는 값은 주석을 보강한다.
- 단순히 이름을 다시 읽어주는 대신, 실제 소비 지점이나 수정 영향을 적는다.

예: 

```dart
/// feature별 top-level route tree를 모은다.
///
/// shell router는 이 목록을 tree 형태 그대로 소비한다.
final appRouteTrees = ...

/// route matching과 current route 판별에 사용하는 flat route 목록.
///
/// route 추가 시 tree 집계와 함께 이 값의 의미도 같이 달라진다.
final appRoutes = ...
```

## 짧은 주석 톤 규칙

- 짧은 설명 주석은 메모형 또는 짧은 설명형을 선호한다.
- 문장을 기계적으로 줄이지 않는다.
- 한 줄로 부족하면 두세 줄로 늘려도 된다.
- 중요한 것은 길이가 아니라, 읽고 나서 "그래서 실제로 뭐 하는 코드인지"가 남지 않는 것이다.
- 파일 헤더와 선언 주석의 톤은 같을 필요가 없다.

## 좋은 상세 주석 기준

### 파일 상단 주석

파일 헤더는 다음 질문에 답해야 한다.

- 이 파일이 담당하는 핵심 기능은 무엇인가
- 이 파일에서 어떤 값이나 정책이 결정되는가
- 이 파일을 수정할 때 함께 확인해야 할 영향은 무엇인가

### 선언 단위 주석

선언 주석은 다음 질문에 답해야 한다.

- 이 선언이 실제로 무슨 역할을 하는가
- 누가 이 선언을 소비하는가
- 이 선언을 바꾸면 어떤 동작이 함께 바뀌는가

다음은 좋은 상세 주석의 예다.

- "feature별 route tree가 여기서 합쳐지며 app 전체 라우팅 구조의 입력이 된다."
- "각 mapper를 순서대로 조회해 첫 번째로 매칭되는 메시지를 사용한다."
- "이 policy를 바꾸면 app 전체 에러 처리 동작이 달라진다."

## 나쁜 주석 기준

- 이름을 그대로 반복하는 주석
- 지나치게 추상적인 설명
- "하지 않는다"만 나열하는 경계 설명
- docs에 있는 구조 설명을 그대로 복사한 주석
- 읽고 나서도 결국 "그래서 실제로 뭐 하는 건데?"가 남는 주석
- 코드와 어긋난 오래된 주석

예:

```dart
/// app 활성 Feature 등록 목록.
final appFeatures = <EngineFeature>[...];
```

위 문장은 composition root에서는 부족할 수 있다.
이 위치에서는 이 값이 어디에서 소비되고 앱 전체 동작에 어떤 영향을 주는지가 더 보여야 한다.

반대로 단순한 데이터 변수나 지역 목록에서는 충분할 수도 있다.

## 좋은 / 나쁜 예시

composition root에서는 부족한 예:

```dart
/// app 활성 Feature 등록 목록.
final appFeatures = <EngineFeature>[...];
```

composition root에서 더 좋은 예:

```dart
/// 이 앱이 실제로 활성화하는 feature 목록.
///
/// 각 feature의 route tree가 여기서 합쳐지며,
/// app 전체 라우팅 구조의 입력이 된다.
final appFeatures = <EngineFeature>[...];
```

나쁜 예:

```dart
/// app root snackbar/dialog에서 사용할 feature error mapper 조합.
String? mapAppErrorNotificationText(Object? domainError) { ... }
```

더 좋은 예:

```dart
/// 전역 에러 알림용 문자열을 결정한다.
///
/// 각 feature가 제공한 mapper를 순서대로 조회해
/// 첫 번째로 매칭되는 메시지를 사용한다.
String? mapAppErrorNotificationText(Object? domainError) { ... }
```

## 예시는 절대 규칙이 아니다

문서에 포함된 좋은 / 나쁜 주석 예시는 방향을 설명하기 위한 것이다.

같은 문장이라도 코드의 위치와 역할에 따라
적절성이 달라질 수 있다.

따라서 예시를 그대로 복사하거나,
특정 형태를 모든 코드에 강제하지 않는다.

항상 다음 기준으로 판단한다:

- 이 코드가 구조적으로 중요한 위치인가
- 이 선언이 앱 전체 동작에 영향을 주는가
- 이름만으로 역할이 충분히 드러나는가

## 주석 언어 규칙

- 코드 식별자 관련 용어는 영어로 유지한다.
- 설명 문장은 한글로 작성한다.
- 구조와 아키텍처 키워드는 영어로 유지한다.

## 한 줄 원칙

모든 주석을 한 줄로 줄이는 것이 목적이 아니다.
클래스와 함수는 기본적으로 설명하고, 변수와 필드는 필요한 곳에 설명을 둔다.
짧게 쓸 수 있으면 짧게 쓰되, 직관성이 떨어지면 여러 줄로 보강한다.

## 최종 기준

주석은 짧게 쓰는 것도, 길게 쓰는 것도 목표가 아니다.
코드의 위치와 역할에 맞는 밀도로 쓰되, 읽고 나서 실제 쓰임새가 바로 이해되어야 한다.
짧게 끝낼 수 있는 곳은 짧게 끝내고, 구조적으로 중요한 곳은 더 자세히 설명한다.
